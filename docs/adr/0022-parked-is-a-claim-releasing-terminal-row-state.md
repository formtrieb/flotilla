# `parked` is a first-class, claim-releasing terminal row state

The engine's fine-state vocabulary gains an **11th state: `parked`** — "deliberately taken out of *this* wave; released for re-planning into a future one." A parked row is **terminal for the wave** (passes the `wave-close` terminality gate, joins `resume()`'s TERMINAL set) and **releases its tracker claim immediately** (spine-first WAL: `spine set-row-state <id> parked`, then `issue-store unclaim <id>`), so the issue returns to the derived `available` pool and a future `wave-plan` draw can pick it up. It is the silent counterpart to the alarm terminals: **`failed`/`abandoned` = terminal + alarm (needs-attention), `parked` = terminal + silence** (no flag, its own close-report rubric).

Found at the M1 live gate (retro P-2/P-5, wave `2026-07-15-wire-contract`): a row deliberately held out of dispatch (its intra-wave blocker never landed) could only pass close as `abandoned` — a state that means "will never be done" for work that *would* be re-planned. The state lied, the claim stayed stuck on the board, and both the hold and the unclaim were manual.

## Why

- **The disposition gap is real and recurring.** `coarse-projection.ts` already promises it: "Terminal-failure states keep the in-flight claim **until dispositioned**" — but the vocabulary had no disposition that means *later*. The only exits were `abandoned` (never) or hand-rolled `unclaim` + a lying state.
- **Mixed-dependency waves are the common shape.** An intra-wave `blocked-by` pair whose blocker slips is not an edge case; holding the blocked row must be scriptable mechanics, not Coordinator improvisation.
- **The claim must not outlive the intent.** A held row that keeps its claim blocks exactly the re-planning it is being held *for*, and lies to humans and concurrent waves reading the board.

## Decisions

1. **Entry edges: exactly two — `planned → parked` and `failed → parked`.** Never from live states (`dispatched`/`report-in`/`reviewing`/`verdict-in`/`re-dispatched`/`approved`/`pr-created`): a running row first resolves through its existing stop paths (`failed`) or completes. This keeps `resume()` free of a "parked with a live worktree" decision column.
2. **Claim released immediately, spine-first.** Set the fine state in the spine (WAL authority, ADR-0002), then `unclaim` on the tracker. `unclaim`'s contract widens from "the reverse edge `queued → available`" to "**any rung → available**" (mechanically it already drops whatever claim exists and is idempotent; the `failed → parked` edge releases an in-flight claim; conformance gains the case).
3. **The coarse projection becomes honest about "no claim": `coarse(state): ClaimRung | null`.** `parked` maps to `null` = "no claim to hold"; the write path executes `null` as `unclaim()`. The single production caller (resume's ledger re-projection) re-projects a parked row idempotently to released — mapping it to any rung would re-claim the issue on resume and re-tell the lie.
4. **Terminal + silence.** `parked` joins `resume()`'s TERMINAL set (decision `keep`; a leftover worktree from a `failed → parked` row is never adopted — no work-carryover promise; a future wave starts fresh from its own anchor, any old branch remains as archaeology). It passes the `wave-close` terminality gate, is **not** flagged needs-attention (the question a flag would raise is already answered), gets its own close-report rubric ("parked — released for re-planning"), and is excluded from the advisory merge-order (no branch, no PR).
5. **Coordinator-set, never event-emitted, never automatic.** No new `WaveEvent`: like `planned`, `parked` is set by the Coordinator (`spine set-row-state` + `unclaim`), and from `parked` every event is invalid (terminal, like `abandoned`). Parking is offered as a scripted **disposition** at exactly two points — `wave-start`'s resolve-blocked-by-against-wave-membership step ("wait within the wave" vs "park") and the STOP disposition menu for `failed` rows ("park for a future wave") — while `wave-create` keeps its default-abort posture on intra-wave `blocked-by` pairs with **no include-and-park**: what you would park at create time does not belong in the wave (do not claim what you cannot run).
6. **No un-park.** Once parked, the claim is released and the issue is back in the pool — a concurrently planned wave may already have drawn it, so a resurrection edge would need exactly the cross-wave claim arbitration flotilla deliberately solves *through* the pool. Re-entry is a **fresh row in a future wave's spine** (states only move forward; the old spine keeps the parked row as history).

## Considered Options

- **Reuse `abandoned` + manual `unclaim`** (rejected — the live-gate status quo): the state semantically lies ("never" for "later"), the release is unscripted handwork, and close flags a deliberate decision as a problem.
- **Name it `deferred` or `held`** (rejected): `deferred` collides with the established **per-gate DoR result** (ADR-0014 — "neither pass nor fail, shows as `deferred`", printed by every `dor` run) and with the M2-deferral prose all over the docs; `held` implies the claim is retained when the whole point is that it is released. `parked` is collision-free and matches the already-established consumer convention ("humans park backlog work in `Backlog`" — wave-setup preconditions).
- **Hold the claim until `wave-close`** (rejected): blocks the re-planning the state exists for and keeps lying on the board — the DES-24 row sat falsely claimed for the wave's whole tail.
- **Map `parked` to an existing rung in `coarse()`** (rejected): `queued` would re-claim on every resume re-projection; keeping the projection total-over-rungs forces exactly that dishonesty. `ClaimRung | null` with one caller is the cheaper truth.
- **Automatic include-and-park at `wave-create` for intra-wave blocked pairs** (rejected): parking is judgment, not mechanics; and a row you already know you cannot run should simply not be claimed into the wave (default-abort stands).
- **An un-park edge within the same wave** (rejected): races the pool — the released issue may be claimed by another wave; forward-only states + pool re-entry is the established arbitration.

## Consequences

- **Engine (build lands in the `parked` build slice, FOR-14):** `ISSUE_STATES` gains `parked` (11); the transition table treats it as terminal (every event invalid, like `abandoned`); `coarse()` signature becomes `ClaimRung | null` with the `null → unclaim` write-path convention; `resume.ts` TERMINAL gains `parked` (+ re-projection `null`); `wave-md-rw` state vocabulary accepts `parked` in Plan-Table cells; `unclaim` doc-contract widens to any-rung with a conformance case (all three stores).
- **Skills:** `wave-close` terminality gate + no-flag rule + the "parked — released for re-planning" report rubric; `wave-start` membership-resolution offers park as a disposition; the STOP disposition menu offers "park for a future wave"; `wave-create` unchanged (default-abort, FOR-8).
- **Docs:** CONTEXT.md — `parked` glossary entry, the Fine-state count 10 → 11, the "every fine state projects to exactly one coarse state" relationship gains the `parked → no-claim` exception, and the `abandoned` boundary is sharpened ("never" vs "later"). CHARTER §6/§11 reconciled.
- **Relates to:** extends ADR-0002 (the projection gains an honest no-claim case; WAL ordering unchanged), sharpens ADR-0006 (parked is deliberately *not* a needs-attention case), ADR-0007 (the state vocabulary is load-bearing — this records the additive change), ADR-0014 (the `deferred` name stays exclusively the per-gate result).
