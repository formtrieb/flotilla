# Linear claims live in workflow states; the triage vocabulary stays in labels

The M2-pulled-forward **`LinearIssuesStore`** maps flotilla's two-scope state onto Linear as follows: the **claim ledger** (`queued → in-flight → in-review`) is written as the issue's **native workflow state** (config-mapped, defaults `Todo / In Progress / In Review`; `done` derives from state *type* `completed` **or** `canceled`/`duplicate` — the same lossy collapse as ADR-0002's `not_planned`→`done`; `unclaim` moves back to `Backlog`). Everything vocabulary-shaped stays a **label**, exactly like the GitHub adapter: the Eligibility OR-set (`ready-for-agent`), the triage vocabulary, `risk/*`/`worker/*`, and the orthogonal `wave/needs-attention` flag (ADR-0006 — a flag *cannot* be a state). `files`/`blockedBy`/`AC` go to the issue body via the **same body-codec** (lifted out of `adapters/github/` to a shared home — it was always markdown-in-body, not GitHub). The store's identity (ADR-0001) is the human team key (`DES-16`); the Linear UUID stays seam-internal. The adapter sits behind a **`LinearApi`** domain seam + in-memory fake (P3 pattern) and must pass the cross-store conformance suite **with zero changes**; the real impl is raw-`fetch` GraphQL behind a **`LinearHttp`** seam (`LINEAR_API_KEY` env, `createLinearApiFromEnv` at the CLI edge — ADR-0019 parity, simpler because Linear has no REST branch).

## Why

- **The board is the point.** ADR-0002's coarse ledger exists so *humans and concurrent fleets see what is claimed*. On GitHub the only projection surface was labels; on Linear the workflow state **is** the surface humans watch — a label-based ledger would leave the board static while agents work, surrendering exactly the visibility the ledger is for. Linear's own GitHub integration moves issues to `Done` on PR-merge, so the platform *already* writes the terminal rung natively; fighting that with labels means two ledgers.
- **The claim rungs and the triage vocabulary compete for one single-select dimension — the claims win it.** ADR-0015 forecast "Linear = workflow-state" for the *triage* dimension, before the claim mapping was decided. Ground truth (the server pilot's `des`/`dev` teams): the standard workflow has no `needs-info`/`ready-for-human` columns, the `started`-type states are needed by the claim ledger, the contract requires the eligibility state's native string to equal the OR-set token, and the consumer's real issues already carry `ready-for-agent` **as a label**. The forecast is corrected by amendment on ADR-0015; the triage vocabulary keeps GitHub parity (labels), with two targeted native touches where Linear is genuinely state-shaped: `closeUnplanned` → the **`Canceled`** state (Linear's `not_planned`), and `applyTriage` clears Linear's Triage inbox (`Triage → Backlog`) as ADR-0004-class best-effort cosmetics.
- **The human-drag hazard is accepted, not engineered away.** `Todo` is a human-used column; a hand-dragged issue reads as `queued` and conservatively shrinks the next wave's candidate pool. Per ADR-0002 the ledger is a lossy projection and the spine is the WAL authority — drift reconciles at `wave-resume`. Consumer convention (the server pilot: humans park in `Backlog`, `Todo` means wave-claimed) + the config-mapped state names keep this honest and portable.

## Considered Options

- **Claims as workflow states, vocabulary as labels, config-mapped state names** (chosen) — board = live wave dashboard; rides the platform's own PR-merge → `Done` motion; portable via `states` config.
- **Claims as `wave/<rung>` labels (full GitHub parity)** (rejected) — collision-free but blind on the board; label-spam is un-idiomatic on Linear; fights the GitHub integration's native state moves.
- **Hybrid: `queued` as label, `in-flight`/`in-review` as states** (rejected) — two mechanisms for one dimension, and the queued rung (the soft claim humans most need to see at planning time) is exactly the one hidden from the board.
- **Triage vocabulary as workflow states (ADR-0015's original forecast)** (rejected) — needs 3+ custom states per consumer team, collides with the claim mapping for the `started` types, and breaks the eligibility-token = label identity the consumer already lives.
- **Native blocked-by relations for `blockedBy`, both directions, now** (deferred, read-side taken) — **read = union** of body-codec ∪ native relations from day 1 (without it the DoR gate is blind to the consumer's existing native relations — a dispatch-too-early correctness hole), **write = body-codec** in this slice; native write is the declared fast-follow (id-resolution + idempotent updates, pure refinement once the adapter stands).
- **Closing probe cross-checking the GitHub API** (rejected) — one seam, one auth; Linear's GitHub-integration attachments already carry the PR merge status. `getClosingState` reads: state-type `completed` + merged-PR attachment → `closed-by-merged-PR`; `completed`/`canceled` without one → `closed-unmerged` (→ needs-attention, ADR-0005); else `open`. Operational preconditions (checked by `wave-setup`): the GitHub integration is installed and wave PRs carry the magic word (`Fixes DES-NN`) — the tracker-dependent close phrase comes from config.
- **UUID as the opaque id** (rejected) — magic words, branch names, spine rows, and humans all speak `DES-16`; ADR-0001 wants the tracker-native *human-visible* identifier. The UUID is a seam-internal lookup detail.

## Consequences

- **`LinearStoreConfig`** joins the `StoreConfig` union: `{ kind: 'linear', team, project?, eligibility?, states?, categoryLabels? }`. `team` is required (owns workflow + label namespace); `project` is the candidate-pool filter (`listOpen` draws only project issues when set — the server pilot binds `des` + `Mock Server`); `states` maps rungs to column names (defaults above); `categoryLabels` maps schema categories to existing consumer labels (the server pilot: `bug`→`Bug`, `enhancement`→`Improvement`).
- **`LinearApi` seam** mirrors the `GitHubApi` shape (create/get/list, label add/remove, comment, state set, relations read, closing probe) with an in-memory fake; the conformance suite runs unchanged against `LinearIssuesStore` — the same zero-change bar `GitHubIssuesStore` passed in P3.
- **body-codec relocates** from `adapters/github/` to a shared adapter home; the GitHub store's imports update, behavior byte-identical.
- **ADR-0015 is amended** (identity row: Linear = labels for the vocabulary; state only for `unplanned` + inbox cosmetics). CONTEXT's Triage-facet and Claim entries update accordingly.
- **The M1 §6 live gate re-scopes to this adapter's consumer**: one real wave on the server pilot's **Mock Server** project — it exercises the Linear tracker adapter *and* the full GitHub git-host side (worktrees, PRs against protected `main`, magic-word close). The wiki pilot's GitHub-Issues live run moves to M2-optional; `RealGitHubApi` stays fixture-proven until a GitHub-Issues consumer materializes.
- **Fast-follows on record:** native blocked-by *write* (the consumer wants it — their existing issues use native relations); `listTriage` batch analytics stays M2 (ADR-0015).
