# `wave-start` records each dispatched row's branch in the spine dispatch-log (the durable branch home is actually written)

At dispatch time, `wave-start` **durably records each row's branch** (and dispatched model, [ADR-0012](0012-worker-is-a-brand-free-autonomy-first-classification.md)) into the spine's `## Resume-Metadata` → `dispatch-log:` block — the section the code already designates *the* durable branch home (`spine-store.ts`: "The dispatch-log is the DURABLE branch home; Plan-Table.branch is derived-only"). This closes a contract-level gap found at the M1 live gate (2026-07-15, the server pilot's Mock-Server wave `2026-07-15-wire-contract`): the branch home was **never populated in a real wave**, silently defeating the resume guarantee ([ADR-0002](0002-two-scope-state-spine-authoritative-one-way-projection.md), an M1 requirement). Three changes make the intent real: (1) `renderSpine` scaffolds the `dispatch-log:` key under `## Resume-Metadata` (it emitted the bare heading only — [ADR-0016](0016-spine-creation-is-an-engine-owned-renderspine.md) said "empty Dispatch-Log section" but shipped no key); (2) a `spine set-branch <spine> <id> <branch> [--model <m>]` CLI verb exposes the already-existing `upsertDispatchLogEntry`/`upsertDispatchLogModel` (present on `SpineStore`, exported from `index.ts`, but wired to **no** CLI verb); (3) `wave-start`'s dispatch WAL calls it — spine-first, in the same step that writes `spine set-row-state <id> dispatched`.

## Why

The `WAVE.md` spine is the authority `resume` reconciles from (ADR-0002/[ADR-0008](0008-spine-stays-local-markdown.md)). `resume()` joins live worktrees to rows **by branch**: it derives the per-row branch from `branchesByIssueId(spine)`, which reads exactly two sources — the Plan-Table `branch` field and the `dispatch-log` entries. The Plan-Table has **no branch column** (branch is a *derived* field per ADR-0016, filled *from* the dispatch-log by `resolveBranches`), so the dispatch-log is the **sole durable home** for the branch. In a real flotilla-run wave that home was empty, because nothing ever wrote to it:

- `renderSpine` emitted `## Resume-Metadata` followed immediately by `## Closed-by` — **no `dispatch-log:` key**. (`upsertDispatchLogToken` *throws* "no dispatch-log: key" without it, so even a caller that tried to record a branch would fail.)
- `wave-start`'s dispatch WAL wrote **only** the fine state (`spine set-row-state <id> dispatched`). The branch (`wave/<id>-<slug>`) reached only the Worker's *prompt*, never the spine.
- No CLI verb wired `upsertDispatchLogEntry`, so the skill layer had no way to record it even in principle.

The consequence is severe and silent: `branchesByIssueId(spine)` returns `{}` → `resume()` computes `branch = null` for every in-flight row → cannot find the row's worktree → decides **`redispatch`** where it should decide `adopt`. A killed coordinator whose Workers had already committed would, on resume, **re-run those rows from scratch and discard the committed work** — the exact failure flotilla's resume design exists to prevent. `wave-close`'s advisory merge-order is degraded the same way (it reads exact branch names from `spine.dispatchLog`).

This stayed invisible through **1116 green tests** because the `resume`/`merge-order` specs feed **hand-authored spines that already contain a populated dispatch-log** (`cli.spec.ts:151`, `resume.spec.ts:39`). The unit path is well-covered; the *real* `wave-start → spine` write is the seam no test crosses. It surfaced only now because this was the first real wave and the first time a live resume would have mattered — the user caught it by noticing the empty `## Resume-Metadata` after `wave-start`.

## Considered Options

- **Write the branch to the dispatch-log at dispatch time** (chosen) — honors the already-decided contract (`spine-store.ts`: dispatch-log is the durable branch home) and ADR-0016's stated-but-unshipped "empty Dispatch-Log section". Smallest change that makes the existing design true: scaffold the key, expose the existing upsert as a verb, call it in the WAL. `resume`/`merge-order` need no change — they already read `branchesByIssueId`.
- **Add a rendered `branch` column to the Plan-Table** (rejected) — contradicts ADR-0016's deliberate "Plan-Table.branch is *derived-only*" and widens an already 9-column table whose cell-escaping is delicate (the `｜` pipe-hardening, [ADR-0019](0019-real-githubapi-is-raw-fetch-rest-graphql-behind-a-github-local-seam.md) fallout). Duplicates a field that has a designated home.
- **Derive the branch by convention at resume time** (rejected as primary) — parse the id out of each live worktree's `wave/<id>-<slug>` branch and match to rows without any durable record. Fragile: it hard-codes the branch-naming convention into the reconciler, breaks if a Worker ever lands on a differently-named branch, and still leaves the model (ADR-0012) unrecorded. Acceptable only as a *defensive fallback* inside `branchesByIssueId` (id-substring match) layered under the durable record — a possible hardening, not the fix.

## Consequences

- **Engine:** `renderSpine` scaffolds `dispatch-log:` under `## Resume-Metadata`; the `renderSpine`/`readSpine` round-trip test is extended so `upsertDispatchLogEntry` on freshly-rendered output no longer throws. A new thin `spine set-branch` CLI verb (id + branch, optional `--model`) routes to `upsertDispatchLogEntry`/`upsertDispatchLogModel` via `SpineStore` (byte-preserving, ADR-0008). No change to `resume.ts` / `merge-order.ts` — they already consume `branchesByIssueId`.
- **Skill:** `wave-start`'s dispatch step records the branch in the WAL, **spine-first**, alongside `set-row-state <id> dispatched` — so a crash after the state write but before the worktree exists still leaves a recoverable branch. `start-mechanics.md` documents the exact invocation; `wave-close` mechanics gain nothing new (merge-order already reads the now-populated log).
- **Regression net:** a **seam test that actually crosses `wave-start → resume`** — render a spine, run the dispatch-WAL sequence (state + branch) through the CLI, then `resume` against a simulated worktree list and assert `adopt` (not `redispatch`). A hand-authored dispatch-log must *not* be the fixture; the point is to exercise the wire that was disconnected.
- **Amends [ADR-0016](0016-spine-creation-is-an-engine-owned-renderspine.md):** its "renderSpine owns the empty … Dispatch-Log … sections" is corrected — an *empty heading* is insufficient; the `dispatch-log:` key must be scaffolded, and (the part ADR-0016 never covered) something must *write* it. The durable branch home is a `wave-start` responsibility, not just a `renderSpine` one.
- **Severity / sequencing:** this is an **M1 blocker** — the "resume is an M1 requirement" guarantee is non-functional until it lands. It does not block *forward* wave execution (dispatch/review/land work; only recovery and merge-order branch-exactness are affected), so a wave can be run and landed today without resume, but M1 §6 acceptance should not be called done until the seam test is green.
- **Follow-on (optional hardening, not required):** add an id-substring fallback in `branchesByIssueId` (worktree `wave/<id>-…` → row) as defense-in-depth, so a future dispatch-log write-miss degrades to convention-match instead of silent `redispatch`. Deferred; the durable write is the fix.
