# A PRD is a tracker document, not a wave issue — published via a Document facet, consumed-state derived from Parent backlinks

A **PRD** is published as a tracker issue but is deliberately **not** a wave issue: it carries no wave Header-Block, no Risk/Worker/Files, and no **Eligibility** marker, so it never enters `listOpen('wave-ready')`. `to-prd` mints it through a dedicated, tracker-agnostic **Document facet** on the `IssueStore` (`publishDocument` / `readDocument` / `listDocuments`) — *not* through `create()`/`CreateInput`, which stay honest as the wave-slice contract. Each slice `to-issues` produces carries an explicit `Parent` backlink in its Header-Block (the PRD's opaque id — [ADR-0013](0013-parent-is-an-opaque-id-reference-not-an-issueref.md)); a PRD's *consumed* status is **derived** from those backlinks (consumed iff ≥1 issue references it as Parent), never written.

## Why

Routing a PRD through `CreateInput` forced fake placeholder `risk`/`worker`/`files` onto a document that has none — and it *still didn't work*: `create()` eligibility-stamps every issue unconditionally, so a PRD with `worker: "HITL-required"` was nonetheless drawn into `wave-plan`'s candidate pool (the "eligibility-pollution" the old `to-prd` skill documented as a known-broken limitation, pushing a filter downstream). "A PRD is a PRD, plain and simple" — the interesting work begins at the slices. Treating the PRD as a tracker document *outside* the wave-slice contract removes the placeholders, the pollution, and the downstream filter in one move, and keeps `create()`/`read()`/`IssueView` fully typed for wave issues (no optional Header-Block).

Deriving *consumed* rather than writing it follows flotilla's existing derive-don't-write discipline (the `available`/`done` bookends, ADR-0002/0003). On GitHub the same `Parent` backlink renders the forward cross-reference on the PRD for free, so a human sees "this PRD was sliced" without any written state or hand-maintained PRD→issue links.

## Considered Options

- **PRD as a tracker document via a Document facet, tracker-agnostic** (chosen) — keeps `to-prd`/`to-issues` tracker-agnostic behind the store seam; the wave-slice contract stays honest.
- **Keep routing the PRD through `CreateInput` with benign placeholders** (rejected) — the smell this ADR removes: fake Header-Block fields, eligibility-pollution, and a downstream `wave-plan` filter to undo it.
- **A dedicated planning-overview skill** (rejected) — we don't want another skill; `wave-plan` surfaces un-sliced PRDs in a separate panel (never as candidates).
- **A written `prd:ready` → `prd:consumed` state flipped by `to-issues`** (rejected) — an extra write with drift risk; *consumed* is derivable from the `Parent` backlinks that exist anyway.
- **`to-prd` goes GitHub-first and bypasses the store (like `triage`)** (rejected) — minimal engine change, but loses the tracker-agnostic property and can't publish PRDs on a MarkdownFs project. The Document facet keeps agnosticism for a bounded cost.

## Consequences

- **Store contract grows:** `IssueStore` gains `publishDocument({title, bodySections}) → id`, `readDocument(id) → {title, body}`, `listDocuments()`. `CreateInput` and `IssueView` gain an optional `parent` (Header-Block field). ~~`parent?: IssueRef` (same mechanics as `blockedBy`)~~ — **superseded by [ADR-0013](0013-parent-is-an-opaque-id-reference-not-an-issueref.md): `parent` is the PRD's opaque id *string*, not an `IssueRef`** (a markdown PRD's `<slug>#prd` id is not `IssueRef`-representable, and `parent` references a document's identity, not a resolvable wave issue). Conformance covers both adapters.
- **`AnnotatePatch` also gains `parent` (amendment; the type is `string` per [ADR-0013](0013-parent-is-an-opaque-id-reference-not-an-issueref.md)).** The decorate path must be able to set the backlink too: a PRD is often realized through a *mix* of newly-created slices and already-filed triage issues that `to-issues` decorates. `AnnotatePatch` shipped (ADR-0010) without `parent` — an oversight relative to this ADR — so a PRD sliced entirely into decorate-targets would never be derived *consumed* (no slice could carry the backlink) and would render no GitHub forward cross-ref. `parent` *is* a Header-Block field, and decorate's job is "add the missing Header-Block fields," so it belongs in the patch; it is written surgically/additively like the other patch fields. `blockedBy` stays **out** of `AnnotatePatch`: changing an existing issue's dependency structure is a larger semantic act than adding wave metadata, and the skill already has the "add it out-of-band first" path.
- **`parent` is a *single* id, not an array** — the slice→PRD graph is a **forest** (each slice has ≤1 PRD home), matching "a slice is sliced *from* a PRD." A slice that genuinely serves two PRDs is a smell to surface, not legalize: split it per-PRD, or promote the shared work to its own (parent-less) issue or its own PRD that both reference. *Consumed* is a coarse human signal, not precise accounting, so single-parent suffices.
- **A PRD is never a `blockedBy` entry.** It is not a wave issue and never lands (`Closes #N` never arrives), so listing it as a blocker would stall the slice's `blocked-by-chain` forever. The **only** slice→PRD relationship is `parent`. The `to-issues` skill states this in one line (the trap is the "blockedBy = what must exist first" reading, and the PRD *does* exist first).
- **Identity / representation:** GitHub uses a `prd` label (not `ready-for-agent`); MarkdownFs uses a `prd.md` beside the slug's `issues/` dir (`.scratch/<slug>/prd.md`), which `listOpen` already never scans; Linear uses a **native Document** ([ADR-0017](0017-linear-document-facet-maps-to-a-native-linear-document.md) — categorically not an issue, so the no-`listOpen` constraint holds structurally). `Parent` is read from the **explicit field, not inferred from the folder path**, so the derivation is identical across stores.
- **`to-prd` rewrite:** publishes via `publishDocument` — no more placeholder `risk`/`worker`/`files`, no eligibility-pollution Limitation note, fewer Common-Mistakes. It supersedes the placeholder-`CreateInput` approach the current skill documents.
- **`to-issues` change:** brings back an explicit `## Parent` per slice (dropped earlier; Matt's template had it as prose) — now a **machine-readable** Header-Block field, which is what makes both the GitHub cross-ref and `wave-plan`'s derivation reliable. Refines ADR-0010 (whose to-issues template listed `## Parent` but no Header-Block).
- **Open / parked (a separate `wave-plan` decision):** `wave-plan`'s PRD overview must aggregate across slug-folders (`.scratch/*/`), since the MarkdownFs store is constructed per-slug; and the multi-person "solo vs team project" scope for scanning claimed issues. Both belong to `wave-plan`'s own contract, not this one.
