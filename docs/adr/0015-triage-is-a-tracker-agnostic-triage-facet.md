# triage writes through a tracker-agnostic Triage facet on IssueStore, not raw gh

`triage` stops shelling `gh` directly and instead drives a dedicated, tracker-agnostic **Triage facet** on the `IssueStore` ŌĆö parallel to the Document facet ([ADR-0011](0011-prd-is-a-tracker-document-not-a-wave-issue.md)): `readTriage(id) ŌåÆ TriageView` ┬Ę `applyTriage(id, {state, category?, comment?})` ┬Ę `closeUnplanned(id, comment)`. The facet is **single-select and intent-shaped** (the adapter computes the native realization ŌĆö GitHub label add/remove, Linear workflow-state set, MarkdownFs status-line), so no GitHub mechanic leaks into the contract. The triage vocabulary becomes a known, typed, overridable `DEFAULT_TRIAGE_SCHEMA` (mirroring `DEFAULT_WAVE_SCHEMA`); the wave-routing core stays **eligibility-blind** ŌĆö it reasons over the triage state-machine *only* through the opaque eligibility OR-set ([ADR-0003](0003-issue-labels-consumer-owned-wave-labels-flotilla-product.md)). The GitHub impl runs through the same injectable `GitHubApi` seam + in-memory fake as `GitHubIssuesStore` (P3), so triage finally gets the conformance-suite guardrail it lacked.

## Why

triage was the one front-half skill that bypassed the engine ŌĆö it shelled `gh issue edit/comment/close` from the skill prose, with no engine guardrail and a hard GitHub coupling. [ADR-0011](0011-prd-is-a-tracker-document-not-a-wave-issue.md) already rejected this same bypass for `to-prd` ("`to-prd` goes GitHub-first and bypasses the store (like `triage`)" ŌĆö rejected: *loses the tracker-agnostic property and can't publish on a MarkdownFs project*) and chose the Document facet. The decision here is to stop making triage the exception and apply that resolution to it too. Three forces:

- **Linear is now a real, near-term consumer, not hypothetical** (decided 2026-06-17). It is already in use on the Ur's own project and is a candidate team standard; the parent project is a likely flotilla consumer. `gh issue edit/comment/close` does not run on Linear at all ŌĆö and the mismatch is not just the CLI but the **representation**: triage's lifecycle state is a single-select dimension that GitHub renders as a *label*, Linear as a native *workflow-state*, MarkdownFs as a *status line*. That representation difference is exactly what the adapter seam exists to hide.
- **The engine deliberately exposes no issue-side label write (ADR-0003).** `transition` writes only `wave/*`. So routing triage through the engine cannot mean "reuse `transition`" ŌĆö it means a *new* facet. A Triage facet that writes the *opaque consumer tokens the skill chose* (never a taxonomy the engine defines) is consistent with ADR-0003's "opaque membership tokens" framing: the **judgment stays in the skill, the mechanical write moves behind the seam.**
- **The states are worth knowing, not just the eligibility subset.** The wave-routing core (state-machine, conflict-map, resume, Worker routing) branches on the triage vocabulary only via the eligibility OR-set ŌĆö it never reasons about `needs-info`/`wontfix`/`ready-for-human`. But flotilla-as-a-whole is *not* blind: triage-funnel / automation-rate / cycle-time **analytics** are a legitimate second reader of the full vocabulary, which is why the schema is known and enumerable rather than opaque tokens.

## Considered Options

- **Tracker-agnostic Triage facet on `IssueStore`, single-select/intent-shaped, universal across adapters** (chosen) ŌĆö mirrors the Document-facet precedent; keeps triage tracker-agnostic behind the store seam; gives it the conformance guardrail; preserves ADR-0003 (facet writes opaque consumer tokens, engine imposes no taxonomy).
- **Keep triage shelling `gh` directly** (rejected) ŌĆö the status quo; loses tracker-agnosticism (no Linear/MarkdownFs triage), no guardrail, and re-instates the exact bypass ADR-0011 rejected for its sibling.
- **Extend the wave engine with a generic `setLabel`/`comment` on `IssueStore`** (rejected) ŌĆö would blur ADR-0003's "two label worlds": a generic issue-side label write on the *wave* contract invites the engine to grow taxonomy awareness. A named *facet* (like Document) keeps the wave contract honest.
- **A separate `TriageStore` adapter, parallel to `IssueStore`** (rejected) ŌĆö a second adapter surface + second conformance suite for a concern that is a *facet of the same issue*. The facet-on-`IssueStore` pattern (already used for Documents) is the cheaper, established shape.
- **`{add, remove}`-style label writes** (rejected) ŌĆö leaks the GitHub representation into the contract; breaks on Linear, where a lifecycle transition is a single-select state *set*, not an add/remove. `applyTriage` is single-select; the adapter computes the swap.
- **Make the Triage facet optional (capability-declared), MarkdownFs defers** (rejected) ŌĆö honest about MarkdownFs having no real triage consumer in M1, and aligned with the capability-conditional lineage of [ADR-0014](0014-dor-is-a-per-gate-deferring-validator-with-a-store-blind-structured-entrypoint.md). Rejected to **keep "facets are universal" unbroken** (Document is universal; introducing the first optional facet is a layering cost not worth saving a thin impl): MarkdownFs gets a thin impl ŌĆö state **reuses the existing `**Status:**` field** (which already holds the eligibility token, so triage-state and eligibility are one field ŌĆö confirmed in the code), category a new `**Category:**` field, comments appended `## Triage Log` blocks, `closeUnplanned` the move-to-`done/`, so conformance symmetry holds.

## Consequences

- **Store contract grows a Triage facet:** `readTriage(id) ŌåÆ TriageView {title, body, state, category, comments[]}`, `applyTriage(id, {state, category?, comment?})`, `closeUnplanned(id, comment)`. Universal ŌĆö both shipped adapters implement it, conformance covers both. `TriageView` is a **separate projection** from `IssueView`: triage state/category never enter `IssueView.status` ([ADR-0003](0003-issue-labels-consumer-owned-wave-labels-flotilla-product.md) holds ŌĆö `status` stays `CoarseState`). It carries the issue's **title + body** (the reported content the agent triages) precisely *because* `IssueView` is wave-header-only ŌĆö that is what lets the skill read the whole report through the facet and drop even `gh issue view`, so "stops shelling `gh` directly" holds for the **read** path too, not just the writes (GitHub already has title+body in hand via `getIssue`; MarkdownFs reads the H1 minus its `NN ŌĆö ` filing prefix + the prose). Analytics reads triage state via the facet, never via `IssueView`; a batch `listTriage(scope)` for funnel reporting is **M2** (analytics is not an M1 requirement), but the known schema makes it an additive add, not a re-architecting.
- **`DEFAULT_TRIAGE_SCHEMA` lands in `contract.ts`** beside `DEFAULT_WAVE_SCHEMA` ŌĆö the 5 default states (`needs-triage`/`needs-info`/`ready-for-agent`/`ready-for-human`/`wontfix`) + 2 categories (`bug`/`enhancement`), overridable in `wave.config`. It is a *default template*, not an imposed taxonomy (ADR-0003): the **shape** of triage (single-select lifecycle + eligibility-marking terminal states) is flotilla's; the **vocabulary** is the consumer's. The skill passes canonical roles; the **adapter resolves** canonical ŌåÆ native (GitHub label string / Linear workflow-state id / MarkdownFs status token) from config ŌĆö the skill never hardcodes a native id. Coherence invariant: the native strings of the eligibility-marking states must equal the `eligibility` OR-set tokens; `wave-setup` enforces this at interview time.
- **`GitHubApi` seam grows** to cover the triage operations (post comment, state-label swap, `close --reason not_planned`, read comment thread) ŌĆö so the GitHub impl is fake-testable like the wave operations, not raw `gh` in skill prose.
- **The AI-provenance disclaimer moves into the facet.** `applyTriage`/`closeUnplanned` prepend `> *This was generated by AI during triage.*` to every posted comment structurally. The small purity cost (the facet "knows" its comments carry AI provenance) buys a guarantee that replaces a forgettable prose Common-Mistake.
- **`ready-for-human` is orthogonal to `HITL-required` ŌĆö and CHARTER ┬¦6's eligibility example was a bug.** Two distinct "human" concepts at two pipeline stages: `ready-for-human` (triage eligibility axis) = **not wave work**, a human handles it entirely outside flotilla, *never* enters a wave; `HITL-required` (Worker axis, [ADR-0012](0012-worker-is-brand-free-autonomy-first.md)) = wave work that *is* surfaced by `wave-plan`, just human-gated. The test that separates them: *does flotilla's wave system track/surface this work at all?* This confirms `public-API-change` is `ready-for-agent` + `background-heavy` (AFK-implementable, landing-gated), **not** `ready-for-human`. Consequence: CHARTER ┬¦6 listing `ready-for-human` as a candidate eligibility-set addition contradicted CONTEXT's "never enters a wave" and is corrected (the example becomes a neutral custom label); CONTEXT gains an explicit `ready-for-human` Ōåö `HITL-required` disambiguation.
- **The `triage` skill is rewritten:** it drops every raw `gh` call for the engine CLI (`issue-store triage-read` / `triage-apply` / `triage-close`), and is restructured to the three-bucket pattern used by `to-issues`/`to-prd` ŌĆö judgment in `SKILL.md`, CLI mechanics in `reference/`, docs (this ADR + CONTEXT) separate.
- **M1 scope grows by the facet** (decided 2026-06-17: "jetzt schon mit zu machen"). The Triage facet + GitHub impl ship in P7 alongside the skill rewrite; the Linear impl is designed-for but built when the Linear adapter lands (still M2 per M1-PRD ┬¦3, but the facet contract is Linear-shaped now). The conformance suite gains triage cases for both M1 stores.

## Amendment ŌĆö on Linear the triage vocabulary is labels, not workflow states (Linear-adapter grill, 2026-07-10)

This ADR's identity row (GitHub=label ┬Ę **Linear=workflow-state** ┬Ę MarkdownFs=status-line) was a forecast made before the Linear claim-ledger mapping was decided. The Linear-adapter grill ([ADR-0020](0020-linear-claims-live-in-workflow-states-triage-vocabulary-stays-labels.md)) corrected it against ground truth: the `started`-type workflow states are needed by the **claim ledger**, the standard Linear team workflow has no columns for `needs-info`/`ready-for-human`, this ADR's own coherence invariant ("the eligibility state's native string must equal the OR-set token") binds `ready-for-agent` to a **label**, and the first Linear consumer (the server pilot) already lives that way. Corrected identity row: **Linear = labels for the triage vocabulary**, with two native state touches ŌĆö `closeUnplanned` ŌåÆ the `Canceled` state (Linear's `not_planned`), and `applyTriage` clearing the Triage inbox (`Triage ŌåÆ Backlog`) as ADR-0004-class best-effort cosmetics. Nothing else in this ADR changes: the facet stays single-select/intent-shaped, and *the adapter computes the native realization* is exactly the property that absorbs this correction without touching the contract.
