# Spine creation is an engine-owned `renderSpine`, not a skill-authored template

A fresh `WAVE.md` spine is produced by an engine primitive `renderSpine(meta, roster, narrative)` (exposed as a `spine create` CLI verb), **not** authored skill-side as a markdown template the way the Ur's `wave-create` did (rendering against `wave-shared/references/wave-md-skeleton.md`). The `roster` is a list of explicit display rows `{ id, title, worker, risk }` — **not** `IssueView[]`, because `IssueView` is wave-header-only and carries no title (ADR-0015); `wave-create` assembles each row from `read(id)` (worker/risk) + `readTriage(id).title` (the Triage facet is the only tracker-agnostic wave-issue title read). `renderSpine` owns **every parser-consumed** section — frontmatter, the Plan-Table (the seven parser-positional columns `ID · Title · Worker · Risk · Reviewer · PR · State`, plus `Iter` and `Reports → Verdicts`), the **`## Conflict-Map`** (parsed by `readConflictMap` and consumed by `merge-order`, so it is rendered from structured `ConflictMap` data in the `CONFLICT_ITEM` format the parser reads), and the empty PR-Log / Dispatch-Log / Closed-by sections. **Only the `## DOR-check` section** — which has no `Spine` field and is consumed by nothing — is injected as an **opaque skill-supplied markdown string** ("A-struct"). `wave-create` shells `dor` (→ opaque DOR-check prose) and `cross-wave` (→ structured conflict data), and passes both in; the engine guarantees every structure `resume`/`merge-order` depend on.

## Why

The `WAVE.md` spine is **parsed** by `readSpine` and is the **authority** `resume` reconciles from ([ADR-0002](0002-two-scope-state-spine-authoritative-one-way-projection.md), [ADR-0008](0008-spine-stays-local-markdown.md)). Yet flotilla shipped no way to *create* one: `wave-md-rw` exposes only mutators — `setRowState` **throws** on a missing Plan-Table row, and the `upsert*` verbs touch log rows, not the roster. So a fresh spine had no engine home — the gap the P7 grill surfaced.

The Ur resolved this **skill-side**: `/wave create` assembled the markdown from a `wave-md-skeleton.md` reference. That is exactly the **dual source-of-truth** flotilla's spine design rejects — the skill's skeleton and `wave-md-rw`'s parser would each independently define the format and could silently drift, with the cost paid on the `resume` path (a malformed spine breaks reconstruction). The byte-preserving `SpineStore` exists precisely so "callers never touch `readSpine`/`setRowState` directly" (ADR-0008); creation belongs behind that same seam. **Parser and printer should be paired in one module** so they cannot diverge.

But not *all* of the spine's sections are equal. `readSpine` parses frontmatter, Plan-Table, PR-Log, Dispatch-Log, **Conflict-Map**, and Closed-by — and the `## Conflict-Map` is consumed by `merge-order` (its `reason` + the ordering), so it too must be engine-rendered in the `CONFLICT_ITEM` format `readConflictMap` reads. The **`## DOR-check`** section is the sole exception: no `Spine` field captures it and nothing consumes it, so its human-facing prose (express-compression, footnotes) stays a skill-supplied opaque string — rendering it in the engine would only drag presentation judgment into a pure function for no drift-safety gain.

## Considered Options

- **Engine `renderSpine` owning every parser-consumed section; only DOR-check injected as an opaque string** (chosen, "A-struct") — render+parse paired on exactly the sections `resume`/`merge-order` depend on (frontmatter, Plan-Table, Conflict-Map); the unconsumed DOR-check prose stays skill-side judgment. Small engine surface, no second format source-of-truth, no `skill-schema-drift` guard needed on the spine.
- **Engine renders the *whole* document, DOR-check included** (rejected, "A-eng") — the engine also formats the DOR-check prose. Couples `renderSpine` to the `dor` result shape and imports the express-variant/footnote presentation logic into a pure function for no drift-safety gain (nothing parses or consumes DOR-check).
- **Skill authors the `WAVE.md` template** (rejected, the Ur's way) — reinstates the dual source-of-truth (skill skeleton vs `wave-md-rw` parser) flotilla designed away; would *require* a `skill-schema-drift` spec to guard the copy — the maintenance cost the engine-owned printer eliminates.

## Consequences

- **`wave-md-rw` gains a `renderSpine` printer (+ an internal `renderConflictMap` paired with `readConflictMap`) + the engine a `spine create` CLI verb.** Input: frontmatter meta (slug, coordinator, model, stamped `Created`/`Last-updated`), the roster as explicit display rows `{ id, title, worker, risk }` (title via `readTriage(id).title` — `IssueView` is header-only, ADR-0015), the structured `ConflictMap` (rendered to the `CONFLICT_ITEM` format `readConflictMap`/`merge-order` read), and the opaque `## DOR-check` string. Output: a full initial `WAVE.md` whose Plan-Table rows are at fine-state `planned` — which `coarse()` projects to the `queued` claim, so they agree by construction. The printer is unit-tested against the same `readSpine`/`readConflictMap` it pairs with (round-trip: `readSpine(renderSpine(x))` recovers the roster **and** the conflict cells).
- **P7.3's engine carryover is `verify?` *and* `renderSpine`** — correcting the grill's earlier "tiny, verify-only" estimate. Still bounded: one optional config field + one printer.
- **Four deliberate de-couplings from the Ur format** (flotilla does not blind-port):
  - **`Reviewer` column is a uniform decoration.** Nothing in flotilla consumes the Plan-Table `Reviewer` column, and flotilla's **Reviewer** is universal (CHARTER §9) — not the Ur's `quick-verify`/`full-review` tiers. So `renderSpine` fills it with a single uniform value; there is **no** Risk→reviewer-profile map (CONTEXT §Risk corrected to match). Distinct from the reviewer **Verdict** that is the AC ground-truth ([ADR-0004](0004-ac-ground-truth-is-the-reviewer-verdict.md)). **(P7.4 grill, 2026-06-19:)** this uniform-Reviewer decoupling is completed in the verdict schema by **dropping `briefProfile` (`quick-verify`/`full-review`)** from `ReviewerVerdict` / `REVIEWER_VERDICT_JSON_SCHEMA` / `validateReviewerVerdict` — it recorded the Ur's Risk→profile selection, which is informationless under a uniform Reviewer. `riskClass` **stays** (it is the load-bearing G3 guard fed to `verdictToEvent`, not a profile tier). Done before the `skill-schema-drift` guard so it cannot cement a vestigial field.
  - **Spine lives at a flotilla-owned path, not `.scratch/`.** Default `.flotilla/waves/<slug>.md` — the Ur's `.scratch/waves/` is the `.scratch/`-as-tracker anti-pattern flotilla rejects. A fixed convention, not a `wave.config` field (the middle-path config stays `{store, verify?}`); promotable to config only if a consumer needs it. Sidecar root → P7.4 / sidecarRoot-in-spine.
  - **No express variant in M1.** The Ur compresses DOR/Conflict-Map when a wave is "express-eligible"; that optimization is YAGNI for M1 — `wave-create` always renders the full form.
  - **Dispatch-order note simplified.** Issue ids are globally unique, so the Ur's "NN-ascending with slug tie-break" filename baggage is dropped: order is blocked-by-topological, else concurrent.
- **`wave-create` is spine-first (the ADR-0002 WAL discipline).** It (1) `renderSpine` + flush the spine, then (2) `issue-store transition → queued` per issue. A crash between leaves the authority (the spine) intact and the claim re-assertable from it; the reverse would orphan a tracker claim with no spine backing.
- **The spine's `## Conflict-Map` carries only *in-wave* conflicts; cross-wave overlaps are a launch-gate, not durable spine state** (skill-half grill 2026-06-18). `cross-wave` returns a `CrossWaveResult` — `intraWaveConflicts` (candidate↔candidate) and `crossWaveConflicts` (candidate↔already-claimed-by-another-wave). Only the `intraWaveConflicts` feed the spine: `wave-create` reconstructs the `renderSpine` `conflict` argument as `{ issues: <chosen ids>, cells: intraWaveConflicts }`, matching "pairwise intersection across **in-wave** issues". The `crossWaveConflicts` are deliberately **not** persisted in the spine — they answer "can this wave launch *alongside the others currently running*?", which is true only at launch time and goes stale as sibling waves close (a durable copy would mislead `resume`). So they are a `wave-create` launch-gate (surface + default-abort when `parallelSafe` is false; the Coordinator may override with an explicit serialization mitigation), never durable spine state. (The path-only `conflict-map` CLI is *not* used to build the spine map — it cannot read a bare GitHub id, M1's target.)
- **Narrative-section content stays a skill responsibility,** so the express-variant logic, footnotes, and conflict-list readability evolve in the skill without an engine change — and a future Linear/MarkdownFs consumer renders the same parsed structure, only the narrative prose differing.
- **(Amended — [ADR-0021](0021-wave-start-records-each-row-branch-in-the-dispatch-log.md), 2026-07-15):** "renderSpine owns the empty … Dispatch-Log … section" was insufficient as shipped. `renderSpine` emitted only the bare `## Resume-Metadata` heading — **no `dispatch-log:` key** — so `upsertDispatchLogToken` threw and a freshly-created spine could never record a branch. ADR-0021 corrects this: `renderSpine` now scaffolds the fenced `dispatch-log:` key (an empty *heading* is not a write target), **and** — the part this ADR never covered — something must *write* it. The durable branch home is a **`wave-start` responsibility** (its dispatch WAL calls `spine set-branch`), not `renderSpine`'s alone; the printer only provides the target the writer fills.
