# triage — facet mechanics

The engine-CLI plumbing for driving the **Triage facet** (ADR-0015). The skill body owns the **judgment** (category, state, what a good brief says); this file owns the **invocation**. Reach for it once you know the outcome you're applying.

> **The CLI is the source of truth for shapes.** Every command prints its usage when run with no args, and validates its input on every call. The JSON below are *worked examples to scaffold you*, not the schema — if one ever disagrees with the CLI, the CLI wins. Don't re-derive validation the engine already does; trust its error and fix the input.

## `{{wave-cli}}` resolution

The wave engine CLI. Your setup pins how it resolves; in-repo that is `npx tsx tools/wave/src/cli.ts`. Every command needs the store config: run from a dir containing `wave.config.json`, or append `--config <path>`. The store (`markdown` or `github`) and the triage vocabulary are selected there — you never name a tracker, and you never write a native label string.

## Commands

| Call | Purpose |
|---|---|
| `issue-store triage-read <id>` | dump the `TriageView` (title, body, state, category, comments) |
| `issue-store triage-apply <id> --input <ApplyTriageInput.json>` | set state and/or category, post a comment |
| `issue-store triage-close <id> --comment <text>` | set the won't-fix state + natively close, in one call |
| any command, no args | usage |

All three run through the same store seam as the wave verbs (`GitHubApi` fake in tests), so triage gets the same conformance guardrail — no raw `gh` in skill prose.

## Read — `TriageView`

```bash
{{wave-cli}} issue-store triage-read <id>
```

Returns the triage projection of the issue — **everything you need to triage the report**, tracker-agnostically:

```json
{
  "id": "42",
  "title": "Login button does nothing",
  "body": "Steps to reproduce: …",
  "state": "needs-triage",
  "category": "bug",
  "comments": [{ "body": "> *This was generated by AI during triage.*\n\n## Triage Notes\n…" }]
}
```

- `title` + `body` are the reported content (the report you triage). They are **not** on `IssueView` — that projection is wave-header-only — so triage reads them here, never via `gh issue view`.
- `comments` is the full thread, oldest-first, **including the reporter's replies** — parse it so you don't re-ask resolved questions.
- `state` / `category` are absent when the issue carries none yet (treat a missing `state` as the entry state, `needs-triage`).

## Apply — `ApplyTriageInput`

One JSON file. The patch is **single-select and additive**: supply only the fields you're setting — the adapter computes the native swap for `state`, and only supplied fields are written.

```bash
{{wave-cli}} issue-store triage-apply <id> --input <outcome.json>
```

The `comment` field holds the **brief body only** — the facet prepends `> *This was generated by AI during triage.*` for you. Never paste the disclaimer into the comment yourself.

**`ready-for-agent`** — state + category + the Agent Brief:

```json
{ "state": "ready-for-agent", "category": "bug", "comment": "## Agent Brief\n\n**Summary:** …" }
```

**`ready-for-human`** — same shape, with the why-not-delegable section in the comment:

```json
{ "state": "ready-for-human", "category": "enhancement", "comment": "## Human-Only Brief\n\n**Why human-only:** …" }
```

**`needs-info`** — state + the Triage Notes:

```json
{ "state": "needs-info", "comment": "## Triage Notes\n\n**What we still need from you (@reporter):** …" }
```

**`needs-triage`** (entry / re-open after a reporter reply) — state only, comment optional:

```json
{ "state": "needs-triage" }
```

A supplied `state` or `category` outside the configured vocab is rejected **before any write** (no partial application) — fix the value and re-run.

## Close — `wontfix`

```bash
{{wave-cli}} issue-store triage-close <id> --comment "polite explanation of why this won't be actioned"
```

One call sets the schema's won't-fix state (`wontfix`) **and** natively closes the issue (GitHub `close --reason not_planned`; MarkdownFs move to `done/`). After it, `triage-read` reports `state: wontfix` and `issue-store read` derives `status: done`. The disclaimer is prepended for you. Don't follow it with a separate close — it's already closed.

## Vocabulary

The triage vocab (states, categories, the entry state, the eligibility set, the won't-fix state) is `DEFAULT_TRIAGE_SCHEMA` in the engine, overridable in `wave.config`. You pass **canonical role names**; the adapter resolves canonical → the store's native representation:

| Concern | GitHub | MarkdownFs | Linear (M2) |
|---|---|---|---|
| state / category | a label | the `**Status:**` / `**Category:**` field | a workflow-state |
| comment | an issue comment | an appended `## Triage Log` block | an issue comment |
| `wontfix` close | `close --reason not_planned` | move to `done/` | the cancelled workflow-state |

You never write any of those native forms — you pass the canonical role and let the store realize it.
